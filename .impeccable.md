# Design Brain — CliffCheck

> **Source of truth for all frontend-design and UI skill invocations.**
> Reference this file before touching any UI code in index.html.

---

## Section 1: Identity

### What CliffCheck Is

A phone-first tool that reveals the hidden math trapping working-class Americans in benefits cliffs — and gives them the exact income target they need to escape. It doesn't lecture. It shows the number. It tells you what to do next.

### Brand Words

| Word | UI/UX Implication |
|------|-------------------|
| **Honest** | Show the real number, plainly. No softening, no jargon. The math is the message. |
| **Empathetic** | Users are stressed about money. The UI should feel like a knowledgeable friend, not a government portal. |
| **Warm** | Stone, amber, natural light. Not cold blue or bureaucratic grey. Physically warm palette. |
| **Clear** | One thing at a time. The cliff chart is the hero. Everything else supports it. |
| **Actionable** | Every screen leads somewhere. The tool doesn't just show the cliff — it shows the way around it. |

### What CliffCheck Is NOT

- A government form (no dense tables, no cold official blues, no legalese)
- A financial planning app (no projections, no investment advice, no accounts)
- A charity product (empowering, not pitying — users are smart people being screwed by a hidden system)
- A data dashboard (CliffCheck is a tool with a single job, not a reporting surface)
- Generic SaaS (no purple gradient hero sections, no "unlock your potential" copy)

### Core Metaphor

A flashlight in a dark room. The cliff has always been there — CliffCheck just shows you where it is so you can step around it. Not magic. Not advice. Just light.

### The "Oh Shit" Standard

Every key output (cliff chart, manager brief) must produce a visible reaction in someone seeing it for the first time. If a judge can scan the chart and immediately understand "this person is $12,000 worse off taking the raise" — that's the bar. If they need to read a legend first, the design failed.

## Section 2: Audience

### Who Uses CliffCheck

Working-class Americans making decisions about income that could cost them thousands. They're on their phones. They're not financial experts. They've probably been told "just take the promotion" by someone who doesn't understand the math.

**Primary:** Warehouse workers, retail staff, gig workers, single parents — anyone navigating SNAP, Medicaid, Section 8, childcare subsidies.

**Secondary:** Social workers, HR professionals, benefits counselors helping clients.

### Three Emotional States

| State | Feeling | UI Must Do |
|-------|---------|-----------|
| "I was offered a promotion and I don't know if I should take it" | Anxious, confused | Lead straight to the calculator. No marketing fluff. Get to the number fast. |
| "I just saw the cliff chart and I'm worse off than I thought" | Shocked, then curious | Don't abandon them at the chart. Immediate cliff alert + path forward. |
| "I need to explain this to my manager" | Nervous, want backup | Manager brief must feel polished and credible — not a screenshot of a calculator. |

### Trust Signals (Required on Every Screen)

Users in financial stress are wary of apps. Two trust signals must be visible at all times:
1. A persistent reminder that data stays on-device ("Your data never leaves this device")
2. No login required, no email ask — ever

## Section 3: Voice & Tone

### Principles

| Principle | Meaning |
|-----------|---------|
| Plain first, precise second | "You're $12,000 worse off" before "Net effective income decreases by $12,000." |
| Name the system, not the person | "This raise triggers a cliff" not "You earn too much for benefits." |
| Agency-first | "Here's your path to $82k" not "You are ineligible above this threshold." |
| Show, don't lecture | Let the chart do the explaining. Copy annotates, doesn't narrate. |
| Concrete numbers, not percentages | "$4,200 in lost SNAP" over "benefit reduction of 34%." |

### Voice by Context

| Context | Tone | Example |
|---------|------|---------|
| Cliff alert | Direct, matter-of-fact | "Taking this raise costs you $8,400/yr in total value." |
| Safe zone | Warm, reassuring | "You clear every cliff at $82k. That's your target." |
| Empty state / pre-fill | Guiding, not demanding | "Enter your current income to see your cliff chart." |
| Manager brief headline | Confident, professional | "Why I'm asking to negotiate this offer." |
| Error / unsupported state | Honest, helpful | "We only have Ohio data right now. More states coming." |
| Trust signals | Quiet, factual | "Your data stays on this device. Nothing is sent anywhere." |

### Approved UI Words

Income, benefits, cliff, take-home, raise, offer, target, safe exit, family size, effective value

### Banned UI Words

Threshold, eligibility, phase-out, net effective income, benefits schedule, FPL, MAGI, subsidy cliff (use "cliff"), ineligible (use "you're above the limit for"), calculation engine, TinyBase, localStorage, state module

### Number Formatting

- All dollar amounts as integers: `$12,400` not `$12,400.00` and not `$12.4k`
- Monthly vs annual: always show annual; show monthly in parentheses below for context
- Negative impact: `−$8,400` (en-dash, red) not `($8,400)`
- Positive impact: `+$8,400` (green)

| Effective income line | — | `#44403C` (stone-700, 2.5px) |

### Colour Rules

- No cold blue anywhere in UI chrome (blue is reserved for the "current income" chart marker only)
- Amber never used as a solid fill for large background areas — warm cream (`#FFF7ED`) for result cards
- Green and red carry financial meaning — do not use for decorative purposes
- The page must feel warm at a glance. If it looks grey or clinical, more amber/cream is needed
- No pure black (`#000`) anywhere — stone-900 (`#1C1917`) maximum

## Section 5: Typography

### Font Stack

| Role | Font | Weight | Size | Usage |
|------|------|--------|------|-------|
| Hero number | Inter / system-ui | 800 | clamp(2.5rem, 5vw, 3.5rem) | Effective take-home totals, cliff delta |
| Section heading | Inter / system-ui | 700 | 18–22px | Card titles, form section labels |
| Subheading | Inter / system-ui | 600 | 14–16px | Field labels, chart annotations, nav items |
| Body | Inter / system-ui | 400 | 14–15px | Descriptions, manager brief prose |
| Caption | Inter / system-ui | 400 | 12–13px | Helper text, trust signals, timestamps |

*Note: CliffCheck loads via CDN/Babel, no custom font loading. Use system-ui / -apple-system stack — it's fast and readable on every phone.*

### Typography Rules

1. **The squint test.** Blur your eyes — the hero number (effective take-home delta) must be the only thing visible. If the chart label and the cliff alert look the same weight, hierarchy is broken.
2. **Numbers are always bigger than their labels.** The label "Effective take-home" is Caption/muted. The value "$58,400" is Hero/primary. Never equal size.
3. **`font-variant-numeric: tabular-nums`** on all dollar values so columns align.
4. **No decorative fonts.** System stack only — this is a tool for people on cheap Android phones with variable font rendering.
5. **Minimum readable size: 12px.** Touch targets 44px+.
6. **Line height:** Hero 1.1 (tight), Heading 1.3, Body 1.6 (generous — this is financial information, not a blog).
7. **Dollar amounts bold.** Any number representing money gets `font-weight: 600` minimum. Money is important.

### Data Display Pattern

Every metric uses a two-layer pattern:

- **Layer 1:** Human label in Caption/muted (`"Your effective take-home today"`)
- **Layer 2:** The number in Hero/bold/coloured (`"$58,400/yr"` in stone-900)

The number is always larger, bolder, and more prominent than its label.

## Section 6: Chart Design

### The Cliff Chart (Primary Output)

This is the most important element in the product. It must be immediately readable without a legend.

**Spec:**
- X-axis: Annual income $0–$120k, labeled every $20k
- Y-axis: Effective take-home (wages + benefit value), in dollars
- Primary line: stone-700, 2.5px stroke, no smoothing (benefit cliffs are sharp, not curved)
- Background zones: colored fills behind the line (green for growth, red for drops, amber for transitions)
- Current income marker: vertical dashed blue line, labeled "You now: $X"
- Offered income marker: vertical dashed violet line, labeled "Offer: $X"
- Safe exit marker: vertical dashed green line, labeled "Safe exit: $X" — only shown when cliff exists
- Chart fills full screen width on mobile, 2:1 aspect ratio, legible at 375px
- X/Y axis labels: 11px, stone-400, no rotation

**Chart annotations (below the chart, not inside it):**
- Cliff alert card (red-50 background): "Taking this offer costs you −$8,400/yr"
- Safe exit card (green-50 background): "Your safe exit income: $82,000/yr"

**What makes a good cliff chart:**
- The drop is visceral — a sharp V-shape that you feel before you read it
- The safe exit is clearly marked — "here's the target"
- The user's current position and offered income are both labeled
- It works in a screenshot (judges will screenshot it)

### Chart Anti-Patterns
- Smooth curves on benefit cliffs (they're step functions, draw them accurately)
- Legend inside the chart (use annotated lines instead)
- Y-axis starting above zero (start at 0 or show full income range)
- Unlabeled markers (every line needs a text callout)
- Chart without the cliff alert summary below it

## Section 7: Layout & Composition

### Single-Column Phone-First

CliffCheck is not a dashboard. It's a single flow: input → chart → output. Layout is a vertical scroll on mobile.

**Primary viewport:** 375px (iPhone SE / budget Android)
**Max content width:** 540px, centered on larger screens
**No sidebar, no nav bar, no multi-column layout** — one column, top to bottom

### Screen Structure (top to bottom)

1. **Header bar** — App name + trust signal ("Data stays on device"). 56px tall. Stone-100 bg.
2. **Input form** — State, family size, current income, offered income. Full-width inputs. No submit button — live update.
3. **Live summary strip** — Single-line rolling result. Updates as user types. Stone-100 bg, full width.
4. **Cliff chart** — Hero element. Full width, 2:1 aspect. Chart.js canvas.
5. **Cliff alert + safe exit cards** — Outcome summary. Two side-by-side cards on wide screens, stacked on mobile.
6. **Manager brief card** — Cream background, readable prose. Copy-to-clipboard action.
7. **Footer** — "Ohio data only. More states coming." Muted, small.

### Spacing — 4px base grid

| Context | Value |
|---------|-------|
| Within form field | 8–12px |
| Between form fields | 16px |
| Between form and chart | 32px |
| Between chart and result cards | 24px |
| Card internal padding | 20px |
| Between result cards | 16px |
| Between sections | 40px |

**Rule:** The chart must have clear visual separation from the form above it. Minimum 32px gap — the chart is the hero, not an extension of the form.

### Input Form Rules

- Full-width inputs on mobile — no side-by-side fields
- No submit button — results update live as user types
- Labels above inputs (not placeholder-only — bad for financial data)
- Currency inputs: format with commas on blur, strip on focus
- The form must not dominate — it's the entrance to the chart, not the product

## Section 8: Component Patterns

### Primary Input Field

```
Label (14px, stone-700, font-600)
Input (full width, 48px tall, stone-200 border, stone-50 bg, rounded-lg)
Helper (12px, stone-400) — optional
```
Focus ring: 2px amber-500. Never blue focus rings.

### Cliff Alert Card (negative outcome)

```
Background: red-50 (#FEF2F2)
Border: 1px red-200
Icon: ⚠ in red-600 (or a simple down-arrow)
Headline: "Taking this raise costs you −$8,400/yr" (16px, stone-900, font-700)
Sub: "You lose $12,600 in SNAP and childcare, plus $3,200 in new ACA premiums." (14px, stone-600)
```

### Safe Exit Card (positive path)

```
Background: green-50 (#F0FDF4)
Border: 1px green-200
Icon: → in green-600
Headline: "Safe exit: $82,000/yr" (16px, stone-900, font-700)
Sub: "At this income you clear every cliff in Ohio. That's your negotiation target." (14px, stone-600)
```

### Manager Brief Card

```
Background: #FFF7ED (orange-50 / warm cream)
Border: 1px stone-200
Padding: 24px
Font: body size (14–15px), stone-700, line-height 1.6
CTA: "Copy brief" button — amber-500 bg, stone-900 text, full-width on mobile
```

### CTA Button (Primary)

```
Background: amber-500 (#F59E0B)
Text: stone-900 (#1C1917), font-700
Height: 48px (44px minimum)
Width: full-width on mobile
Border-radius: rounded-lg
Hover: amber-600 (#D97706)
```

No cold blue buttons anywhere.

### Trust Badge (Persistent)

```
Background: stone-100
Border-bottom: 1px stone-200
Text: "🔒 Your data stays on this device — nothing is sent anywhere" (12px, stone-500, centered)
```
Shown at top of page, always visible.

## Section 9: Motion & Animation

### Philosophy

No ambient animation. This is a financial tool for stressed people — motion should feel calm, not playful. The only sanctioned motion is the chart drawing on first load and the live summary strip updating on input.

### Rules

- Form inputs update results immediately (no loading states on input change)
- Chart redraws on income change — smooth 200ms transition max
- No skeleton loaders (data is computed locally, instant)
- No number ticker animations on results — financial numbers need to be read accurately, not watched count up
- No entrance animations on cards
- Hover states only on interactive elements — never on static text or the chart

## Section 10: Anti-Patterns

### NEVER (Non-negotiable before any UI build)

- [ ] Use cold blue (`#3B82F6`, `#2563EB`) anywhere except the "current income" chart marker
- [ ] Use a submit/calculate button — the form updates live
- [ ] Show raw annual dollar amounts without the `/yr` label
- [ ] Use a percentage when a dollar amount is available
- [ ] Render the cliff chart with smooth bezier curves (cliffs are step functions)
- [ ] Nest cards inside cards
- [ ] Build a multi-column layout at 375px
- [ ] Use amber as a solid card background (cream `#FFF7ED` is for card fills)
- [ ] Show a chart without the cliff alert and safe exit cards below it
- [ ] Ship an empty manager brief — if there's no cliff, say "No cliff detected. This raise improves your total value."
- [ ] Use font-mono on labels or descriptions
- [ ] Render dollar values without `font-variant-numeric: tabular-nums`
- [ ] Place trust signal below the fold
- [ ] Build any screen that requires explanation before it's understood

### Named Anti-Patterns

| Name | What It Means |
|------|---------------|
| "Government Form" | Dense tables, official blues, legalese labels, submit buttons |
| "Pity Design" | Soft, apologetic tone that treats users as victims rather than smart people being screwed by a hidden system |
| "The Hidden Cliff" | Chart without cliff zone annotations — the drop is visible but nobody told the user what it means |
| "Numbers Without Stakes" | Showing raw values without the impact: "$58,400" means nothing without "that's $4,200 better than taking the raise" |
| "Percentage Dodge" | Using "34% reduction" instead of "$4,200 loss" — concrete dollars are always more powerful |
| "Cold Blue CTA" | Default Tailwind blue buttons — they read as government/clinical, not empowering |
| "Submit Culture" | A calculate button instead of live-updating results — the user should never have to ask for the answer |
| "Chart Without Context" | Rendering the chart without the cliff alert and safe exit summary below it |
| "Apology Empty State" | "Ohio only, sorry" instead of "Ohio is fully supported. California and Texas coming next." |
| "Equal Weight" | Form labels and result values at the same font size — the number must be visually dominant |

## UI Build Toolchain

For any frontend task:

1. **Read this file first** — palette, principles, and anti-patterns before writing a line of code
2. **Build in index.html only** — no separate files, no build step, CDN imports only
3. **Test at 375px** — open DevTools, set to iPhone SE. This is the primary viewport.
4. **The "Oh Shit" test** — show the cliff chart to someone unfamiliar. Can they read the impact in 5 seconds without explanation? If not, redesign.
5. **Check the anti-patterns checklist above** before marking any task done.
